FIS是专为解决前端开发中自动化工具、性能优化、模块化框架、开发规范、代码部署、开发流程等问题的工具框架。

专注于前端构建，无需进行任何配置轻松应付常见需求。个问题：让残障人士能无障碍地访问网页上的动态内容。

前端工程师都在做什么？

在聊fis之前，让我们先来看看前端工程师们，成天都在干些什么

说起前端工程师的工作，大致分为：切图、html、css、js，要求多一点：图片拼接（css sprites）、css压缩、js压缩，再要求多一点：代码重用、语义化、自适应、模块化……

接下来让我们来简单看看前端代码结构的演化过程（不一定精确，意思意思）

最早期的页面，html、css、js混杂在一起

<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Strict//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-strict.dtd">
<html>
    <head>
    </head>
    <body>
        <div style="width:100px;height:100px;" onclick="alert(1)"></div>
        <font size="12px">font</font>
    </body>
</html>

然后，人们认识到了这种代码极不易维护,于是写成如下模式

<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Strict//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-strict.dtd">
<html>
    <head>
        <style>
            .div{width:100px;height:100px;}
            .font{font-size:12px;}
        </style>
        <script>
            function foo() {
                alert(1);
            }
        </script>
    </head>
    <body>
        <div class="div" onclick="foo();"></div>
        <div class="font">font</div>
    </body>
</html>

再然后，人们觉得这样页面提交还是太大, 干脆来个css、html、js完全分离

/* style.css */
.div{width:100px;height:100px;}
.font{font-size:12px;}

// foo.js
function foo() {
    alert(1);
}

<!-- html --> 
<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Strict//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-strict.dtd">
<html>
    <head>
        <link rel="stylesheet" href="style.css" />
        <script src="foo.js"></script>
    </head>
    <body>
        <div class="div" onclick="foo();"></div>
        <div class="font">font</div>
    </body>
</html>

前端就这些事？当然不是！当做一个特别简单的网站的时候，前面说的这些就差不多了。但是当网站做到一定规模，代码量十分庞大，功能交互非常复杂，对性能要求很高的时候，问题就渐渐出来了……

那么问题就来了

  • 图片icon：
    网站一般都会用的很多小图标，一开始我们是切成一个个小小的图片，然后写进css。可是后来发现数量庞大的小图片会发出很多次请求，大大滴干扰了服务器的性能。于是css sprites出现了，就是将一个个小小的图片拼接成一个的图片，然后在css中通过‘背景位置’来定义一个个小图标，这样许多图标就只需要发出一个请求。
    那么问题来了，每次更换图标都有拼接，还要去量一个个图标的坐标，累死人了有木有……

  • 资源压缩合并：
    随着网站的复杂度增加，代码量也在增加。js文件从一个请求到几十个请求，从1k到几百k甚至几M。网络的负担很重。怎么办呢，可以讲多个js合并为一个js，然后进行压缩，将空格注释去掉，将长单词换成短单词。
    那么问题来了，怎么拼？怎么压？

  • 代码重用：
    一个网站有很多个页面，其中会有很多相似的功能，难道我们要每一个写一遍？当然不是，此时就需要代码重用，将公用的部分提出来，需要的时候调用就行了。
    那么问题来了, 每个页面需要的功能各不相同，怎么取组织引入重用的代码？

  • 资源依赖：
    这是在前面代码重用的基础上说的，如果你将所有代码放在一个文件，那就还是别谈了……
    ‘index.html’可能依赖一个’index.js’和’index.css’，’index.js’依赖’global.js’, ‘index.css’依赖’global.css’……当然实际情况比这复杂的多。
    那么问题来了，我怎么知道哪个依赖哪个？又在什么时候、怎么引入它们？

  • 资源嵌入：
    有时候我们需要将各个页面的公用部分（比如：header.tpl）提取出来，然后在需要的地方嵌入，有时候需要将img转换为src:base64的格式而不是url。
    那么问题来了，怎么嵌入？ajax（切！）? 后端代码（滚蛋，我们在讨论前端）？或者……？

  • 资源定位：
    你有木有这种困惑，就是页面引入一张图片或者css、js的时候，该怎么定义它们的路径？绝对路径？相对路径？
    如果我们用根路径（‘/’）,突然网站从一级目录搬到了二级目录怎么办(比如网站根路径从’http://yourdomain.com‘ -> ‘http://yourdomain.com/myweb/')？
    用相对路径？css里的路径是相对css的，js的路径是相对引用它的html的，很乱有木有，说说都晕了。
    更高级的时候，有可能发布后的目录结构和开发的目录结构不一样（比如图片统一放到静态资源服务器）
    那么问题来了，肿么办呢？

  • 版本号管理
    为了解决浏览器缓存问题，我们常常需要给静态资源文件添加一个版本号。
    那么问题来了，肿么加？一个个加？通过后端代码加（我们只谈前端好吧）？

解决方案

针对上诉几个问题，也许每个团队都有自己的方案，也出现过许多不错的并且名声远扬的方案。

  • lab.js
    早起的js加载方案，可以用来动态加载js

  • require.js sea.js esl.js
    js模块化解决方案，不管是cmd还是amd，都大同小异，用来解决模块依赖和按需加载

  • grunt.js
    Grunt是基于Node.js的项目构建工具。它可以自动运行你所设定的任务。Grunt拥有数量庞大的插件,几乎任何你所要做的事情都可以用Grunt实现。

  • gulp.js
    新生代流式（pipe）构建工具，最近风头正盛

  • Bower，Component，npm，Browserify
    各种包管理工具

  • FIS
    真正的前端解决方案

这么多解决方案，我为什喜欢FIS？

个人认为，前面几种解决方案，却是都很不错，但是总感觉他们都是只能解决前端开发中的一部分问题。比如最近很火的gulp，基于流式的构建的确很新颖，使用起来也很爽。
不过，好多东西实现起来很困难（有人也许会反对，说它无所不能……），比如我要公共组件引用嵌入、自动css sprite、变目录结构是的资源依赖等等……虽然用各种各样的插件好像每个功能都能实现，但是总觉得缺乏一种系统的感觉……
上面是个人的浅见，因为本人使用gulp的时间的确不多，所以不喜勿喷……

那么，为什么使用FIS，因为我发现它的理念很先进，并且能实现几乎一切我想实现的功能。

如若说其他都是“前端工具”的话，FIS就是真正的“前端解决方案”
